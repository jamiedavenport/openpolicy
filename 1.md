## 0. The one-sentence thesis

v1.0 is the moment we **lock a public contract**, **rename into the PolicyStack
brand**, and **fold OpenPolicy + OpenCookies into one config object** so that a
single declaration drives the rendered policy *and* a jurisdiction-correct
consent runtime — while keeping the architecture we already have, because on
inspection it is sound.

"v1.0 can be a rewrite" — after reading both codebases, **don't rewrite.** Both
engines are well-factored and fit-for-purpose. The work is concentrated in
four places: the **scanner's delivery mechanism**, the **agent/DX surface**,
the **absence of a frozen public API**, and — newly confirmed by the audit —
**reconciling two jurisdiction models, one of which is missing its core
behavior**.

## 1. Strategic context

Not a greenfield 1.0. The brand migration is already in flight and there are
**three repos**, not one:

- `/Users/jamie/openpolicy` — the policy engine (`@openpolicy/*`, 8 packages).
- `/Users/jamie/terms-sh/opencookies` — the consent runtime (`@opencookies/*`,
  10 packages). Same toolchain (Vite+ `vp`, pnpm, changesets, oxc-parser).
- `/Users/jamie/terms-sh/policystack.dev` — the docs + marketing site, a fresh
  **TanStack Start** rewrite (Vite 8 + Nitro + React 19 + Tailwind v4,
  content-collections MDX, Pagefind, Shiki). Already hosts docs for *both*
  products (`content/docs/{openpolicy,opencookies}`), a live `/llms.txt` route
  + `.md` route variants, and **dogfoods `@openpolicy/react`/`@openpolicy/sdk`**
  to render its own `/privacy`. It markets PolicyStack Cloud (the hosted tier) too.
  `openpolicy.sh` is already a redirect shim → policystack.dev.

All three repos fold into **one PolicyStack monorepo** at v1.0.

Hosted tier already exists (`@openpolicy/plus` `client.consent()`, offstage.sh).
Project memory is partially **stale**: `packages/vite-auto-collect` and
`packages/astro` do not exist — the scanner lives in `@openpolicy/vite`
(`analyse.ts`/`scan.ts`); Astro consumes `@openpolicy/core` directly.

**Implication:** the product identity is *PolicyStack, the headless privacy
stack*. The `Open*` names do **not** survive as nested sub-brands — they are
*renamed*, not contained. Ship v1.0 under `@policystack/*` so npm scope, docs
domain, product names, and repo all align at one breaking boundary (decision 6):

| Was | Becomes | Layer |
|---|---|---|
| OpenPolicy | **PolicyStack Policy** | OSS — the policy engine (policy surface of `@policystack/*`) |
| OpenCookies | **PolicyStack Consent** | OSS — the consent runtime (`@policystack/consent` + the `./consent` subpaths) |
| PolicyCloud / offstage.sh | **PolicyStack Cloud** | Hosted — diffing, the consent ledger, hosting, change notifications (out of OSS scope) |

"OpenPolicy"/"OpenCookies"/"PolicyCloud" survive only as deprecated npm scopes
with `console.warn` shims (§10) and in the GitHub repo *history* — the repo is
kept and **renamed in place** to preserve stars/issues (§12 phase 0) — never in
v1.0 docs, READMEs, package names, `llms.txt`, the skill pack, or marketing.

### The OSS / hosted line (decision 3)

OSS scope only. Hosted (PolicyStack / offstage.sh) owns and OSS must **not**
reimplement: policy version *diffing* (OSS emits a stable `version` hash; the
*diff* is hosted), the proof-of-consent ledger, hosting/CDN, change
notifications, audit exports. OSS's job: be the deterministic, type-safe engine
with **stable extension points** (§9) the hosted layer plugs into.

## 2. OpenPolicy — assessment

### 2.1 Config model — *good, lock it*

`OpenPolicyConfig` (flat input) → discriminated `PolicyInput` → compiled
`Document`. The data/cookies shape is symmetric (`{ collected, context }` /
`{ used, context }`) with the 1:1 key constraint enforced at the *type* level
by `defineConfig`'s generics. This is the asset to freeze, not change.

Fix before freezing: (1) **validation is implemented twice**
(`validate-config.ts` on flat input vs `validate.ts`/`validate-cookie.ts` on
expanded) — consolidate to one validator with a single documented `IssueCode`
union, since codes become public API at 1.0; (2) **do not prune
`Jurisdiction`** — land the jurisdiction capability table (§4.2.1) instead:
keep breadth, but every under-backed jurisdiction emits a *located diagnostic*
rather than a silent weak policy; (3) add **strict mode** (warnings→errors + per-code suppression);
(4) document that user strings pass through untranslated.

### 2.2 The scanner — *keep the scan, change the delivery* (decision 4)

The scan is solid (oxc-parser, two-phase binding→call walk, handles renamed
imports). The fragility is the **delivery**: intercepting the SDK's internal
`./auto-collected` import via a `\0virtual:` module — the root of the
`optimizeDeps.exclude` (OSS-7/#57) and `ssr.noExternal` (OP-170) workarounds,
full-reload-only HMR, and **silent data loss** (dynamic category, const-extracted
labels, import alias, spreads → dropped with zero diagnostics).

Hardening (the key technical recommendation):

- Replace virtual-module interception with a **first-class on-disk generated
  module** (extend the existing `openpolicy.gen.ts` to also carry the data,
  imported explicitly). Deletes the pre-bundle / SSR-externalize /
  `optimizeDeps` fragility tiers at once; HMR becomes normal invalidation.
- **Diagnostics, not silence:** every skipped *recognized* call emits a
  located build warning (the scanner already knows the call site before it
  `return`s).
- Resolve the SDK specifier through the resolver, not exact string match
  (handles aliases + the `@openpolicy`→`@policystack` rename window).

### 2.3 Document AST + renderers — *sound; do not adopt unified/mdast*

~15 hand-rolled node types with a domain-specific `context.reason` compliance
trace. We *emit* a closed legal document; we don't *parse* arbitrary Markdown —
mdast/unified would add surface and footguns for zero benefit. PDF via `pdfkit`
(not react-pdf/puppeteer) is right. Real debt = **6 duplicated tree-walkers**
(md/html/pdf/react/vue/svelte): consolidate to one `visit()` in core + per-
renderer visitor maps. Also: `OutputFormat` includes `jsx` which **throws** —
remove it from the public union (React/Vue/Svelte *are* the JSX story).

**AST review — a pre-freeze gate (Note 2 / decision 8).** Keeping the
hand-rolled AST is decided; its *shape* is not yet frozen and `Node` + `visit()`
are on the 1.0 frozen surface (§6), so there is exactly one shot. Run a
deliberate pass before phase 6 — output a short ADR + any breaking node changes
landed in **phase 2**, never after the tag:

- **Discriminant hygiene:** one consistently-named `type` tag on every node, no
  structural-only variants — this is what makes the single `visit()` + per-
  renderer visitor maps total and exhaustively type-checkable.
- **Taxonomy:** are ~15 the right cut? Collapse near-duplicate nodes; split any
  node a renderer must branch on internally (a branch in a visitor map = a
  missing node type). Confirm no node leans HTML — names stay legal/semantic
  and renderer-agnostic (also targets PDF and native component trees).
- **`context.reason` compliance trace (the differentiator):** make it a
  first-class *typed* field linking back to the `JurisdictionId` /
  `IssueCode` / `lawfulBasis` that produced the node — not a free-string side-
  channel. The hosted diff/audit layer (§9) keys off it; it must be stable and
  machine-readable.
- **Serialization & forward-compat:** the AST must be JSON-round-trippable
  (hosted diffing / `version` hash, §9) and carry a small `astVersion` + an
  unknown-node story so the frozen union can gain node types post-1.0 without a
  major bump — semver locks this at 1.0, so design it now.
- **Consent-derived output:** the merge means some rendered text is consent-
  derived (the §4.2 posture statement). Decide *before* freezing whether that
  is new node types or composed existing ones — retrofitting nodes post-1.0 is
  the expensive case.

### 2.4 Framework integrations & CLI

React/Vue/Svelte slot maps have drifted (Vue missing `Root`/`ListItem`);
canonicalize the slot contract once in core. CLI weakest area — see §8.

### 2.5 What the user should *not* have to write (Note 3 / decision 9)

The sharpest framing for the whole product: **the config is the *legal* layer;
the code is the *technical* layer; nothing technically knowable is hand-
declared.** A scanner can prove *what* data flows and *where* it goes; only a
human can state *why* (purpose), *under what basis* (`lawfulBasis`), and *for
how long* (retention). The rule, as doctrine:

> Derivable from source → a call-site helper + the gen module.
> A legal judgement → stays declarative in config.
> The validator polices the boundary (a scanned key with no legal context is a
> *located* error, not a silent weak policy — already the §2.1 generic).
> Default to opt-in / explicit: call sites are greppable and diffable, so the
> disclosure surface is itself auditable (git history = the disclosure log).

**Remove / stop asking:**

- The `...dataCollected` / `...thirdParties` / `...cookies` **spread
  boilerplate**. Today the user hand-spreads scanner sentinels into the config.
  The on-disk gen module (§2.2/§7) makes that an implementation detail —
  `defineConfig` should *merge the gen module itself* so the user never writes
  the spreads. (Keep the typed augmentation that forces sibling legal context;
  drop only the manual spread.)
- `consentMechanism: { hasBanner, hasPreferencePanel, canWithdraw }` — facts
  about the *wired runtime*, not legal judgements. §4.1 already proposes
  validating them against what's wired; go further and **derive** them
  (provider present ⇒ `hasBanner`; preferences route present ⇒
  `hasPreferencePanel` / `canWithdraw`). Three fields gone — and they can no
  longer be a lie.
- Seed `company.{name,url,contact}` from `package.json`
  (`name`/`homepage`/`author.email`) as overridable defaults — not inferred
  truth (legal entity ≠ npm name), just a smaller blank config.

**New call-site helper (same frozen, syntactically-scanned shape as
`collecting`/`thirdParty`/`defineCookie`, §6):**

- `sharing(key, recipient)` — marks data *leaving* to a third party at the
  point it leaves. The missing primitive for the CCPA/CPRA "sell/share"
  determination that drives the §4.2 opt-out posture and the §4.3 declared-vs-
  used cross-check. Highest-value addition: it ties the opt-out legal posture
  to a provable code fact, not a checkbox.
- `processing(key, { region })` — **deferred post-1.0 (decided).** Cross-border
  transfer disclosure overlaps config-declared sub-processors and is not worth
  freezing a call shape we're unsure of; `sharing()` is the only *new* frozen
  helper at 1.0. Revisit post-1.0 with real demand (YAGNI on frozen surface).

Net: the v1 hand-written config trends toward *only* the irreducibly-legal
fields (`company`, `jurisdictions`, and the per-key `purpose / lawfulBasis /
retention / provision` context); everything else is inferred, derived, or
seeded, and the validator guarantees the two halves agree. **Decided (decision
12):** privacy and cookie remain *two separate documents* — the
`type: "privacy" | "cookie"` discriminant stays; the merge unifies the
*config/consent* story, not the documents. The AST keeps two document roots and
the §2.3 review treats them as distinct, never as a merge target.

### 2.6 Translations going forward (Note / decision 14)

The current model, on inspection, is *good* and mostly stays: a closed
`Locale = "en"|"fr"|"de"|"nl"|"es"`, ~125 emitted strings, an **exhaustive
typed-function `Dictionary`** where `tsc` is the lookup (no runtime path
resolver) — adding a key that a locale misses is a compile error, not a silent
English leak. For a *legal* document that completeness guarantee is the whole
point: an unknown locale must **not** silently fall back to half-English. So:

- **`Locale` stays closed and on the frozen surface (§6).** Do not widen it to
  an open `string`. The exhaustive `Dictionary` contract is the feature.
- **Add a custom-`Dictionary` extension point (§9).** `compile`/`createT`
  accept an optional caller-supplied `Dictionary` (same exhaustive type). This
  decouples "ship a language" from "cut a core release / widen the frozen
  union" — the §11 post-1.0 i18n-scaling worry — and the consumer inherits the
  same `tsc` completeness guarantee, so a bring-your-own locale cannot be
  half-translated either.
- **The i18n boundary is the §2.5 doctrine again:** PolicyStack translates *the
  words it writes* (the scaffolding `Dictionary`); never *the words you write*
  (`purpose`/`retention`/company strings — pass through untranslated, already
  the §2.1 note) or machine metadata (`reason`, section IDs, validator
  messages — stay English, developer/tooling-facing). Frozen contract.
- **Consent reconciliation (closes the §11 flag).** The merged single config
  carries one `locale: Locale` (closed, shared by policy *and* consent). The
  consent *record* keeps `locale: string` at rest for legacy/forward record
  tolerance (it already does `fromUnknown` legacy parsing) but is *written*
  from the canonical `Locale`. `locale` is already in the policy-version hash,
  so "visitor switches language → version changes → re-prompt" falls out for
  free via the same §4.1 mechanism — and is *correct*: different-language
  visitors are shown materially different policy text and should re-consent.
- **Migration.** OC accepts any `locale` string today; tightening to closed
  `Locale` is technically breaking. Handle with the same warn-and-map shim as
  the §4.2.1 aliases / §10 deprecation — shipped in the `0.x` consent window,
  removed before the freeze.

## 3. OpenCookies — audit findings

**Verdict: architecturally mature and consistent, but explicitly pre-1.0 and
missing one piece of core behavior** (the jurisdiction question, §4.2). It is
*not* the liability §11 of rev 1 feared — it's a well-built engine that needs a
deliberate config-bridge and one real feature before its API can join the
frozen 1.0 surface.

What's strong:

- **5 framework adapters** (React, Vue, Svelte/runes+stores, Solid, Angular) +
  an Astro example on vanilla core. The public API is *semantically identical*
  across all five (`Provider`/setup, `useConsent()`, `useCategory(key)`,
  `ConsentGate`), differing only by framework reactivity idiom. No drift.
- **Headless core** ≈4kb: a clean `ConsentStore` (`getState`, `subscribe`,
  `acceptAll/acceptNecessary/reject/toggle/save`, `setRoute`, `has`,
  `refreshJurisdiction`), a composable `ConsentExpr` (`and/or/not`) for script
  gating, three solid `StorageAdapter`s (localStorage / cookie-with-SSR-`Set-Cookie`
  / server-endpoint), and a `vendors.json`-driven scanner.
- **Versioned consent records** (`ConsentRecord { schemaVersion:1, decisions,
  policyVersion, decidedAt, jurisdiction, locale, source }`) with re-prompt
  triggers `policyVersion | categoriesAdded | expired | jurisdiction` and an
  `oncookies:reprompt` CustomEvent. Legacy-tolerant record parsing.
- **Script gating** with pre-built integrations (GA4, Meta Pixel, PostHog,
  Segment) that stub global queues until consent then replay.
- Clean code: **no TODO/FIXME/experimental markers**, 11 test files / ~2.5k
  test lines.

What's not 1.0-ready:

- Versions `0.0.x`; README says "APIs may change before v1."
- Changeset config is **independent** (not fixed/linked) — opposite of
  OpenPolicy's fixed group.
- `@opencookies/cli` is an **empty scaffold** (`export {}`); its `apps/docs` is
  a placeholder whose "Fumadocs/OP-294" plan is **superseded** by
  policystack.dev — delete it on fold-in.
- The `Category` type carries no lawful basis / vendor / purpose metadata —
  the impedance mismatch with OpenPolicy (§4.3).
- **The opt-in/opt-out jurisdiction divergence is not implemented in core**
  (§4.2) — the headline finding.

## 4. The merge — the headline 1.0 narrative (decision 2)

The compelling 1.0 claim: **"declare your privacy story once; get the policy
*and* a jurisdiction-correct consent banner that can't drift from it."** Today
the two are composed by hand and configs are *not* shared. Closing that is the
flagship work.

### 4.1 The shared-config bridge

`OpenPolicyConfig` already has the hooks:

- `cookies: { used: { essential:true, [k]:boolean }, context: { [k]:{ lawfulBasis } } }`
- `consentMechanism: { hasBanner, hasPreferencePanel, canWithdraw }` — today
  this only *describes* consent in the rendered policy text.

`OpenCookiesConfig` wants:

```ts
{ categories: { key, label, locked?, description?, respectGPC? }[];
  policyVersion?; locale?; initialRoute?; jurisdictionResolver?;
  gpc?; adapter?; triggers? }
```

**1.0 move:** make the merged config the single source of truth. `defineConfig`
output *derives* the consent runtime:

- `cookies.used` keys → `categories[]`. `essential:true` → `locked:true`.
- `cookies.context[k].lawfulBasis` → the per-category posture
  (`legal_obligation`/`legitimate_interests` ⇒ no consent gate;
  `consent` ⇒ gated). **This is why `lawfulBasis` must move onto the consent
  `Category` (or be threaded via a bridge) before the consent API can freeze.**
- OpenPolicy's `version` hash → `policyVersion` → automatic re-prompt on policy
  change. These concepts line up *exactly* and are the cleanest part of the
  merge — wire them and "policy changed, re-consent" is free.
- `consentMechanism.canWithdraw` → the preferences-route affordance;
  `hasBanner/hasPreferencePanel` validate against the runtime that's wired.

Impedance to resolve deliberately (not heuristically): the jurisdiction enum
mismatch (`eu`↔`EEA`, `us-ca`↔`US-CA`) is **removed outright** by the single
canonical `JurisdictionId` (§4.2.1), not bridged; `cookies.used[k]:boolean`
default-state vs `locked`; no vendor/purpose on `Category`. Specify an explicit bridge
type — do **not** ship the lossy `lawfulBasis==="legal" ? locked` heuristic.

### 4.2 The jurisdiction question — opt-in (GDPR) vs opt-out (CCPA)

You asked specifically how OpenCookies handles this. The honest finding:

> **It ships the plumbing but not the policy.** OpenCookies has a rich
> jurisdiction *resolution* layer but **does not encode opt-in vs opt-out
> default state in core**. Every jurisdiction gets the *same* initial state:
> `decisions = Object.fromEntries(categories.map(c => [c.key, c.locked===true]))`
> — i.e. non-locked categories default **off** everywhere. The README's
> "opt-in for EEA/UK, opt-out for US" is **delegated to whoever writes the
> banner UI**; core does not do it.

What *is* implemented and is genuinely good:

- `Jurisdiction = "EEA" | "UK" | "CH" | "US" | \`US-${string}\` | "BR" | "CA" |
  "AU" | "ROW"`, with `countryToJurisdiction()` and **four pluggable
  resolvers**: `headerResolver()` (cf/vercel/netlify edge headers),
  `timezoneResolver()`, `manualResolver()`, `clientGeoResolver()` (the only one
  that refines US → `US-CA` etc.).
- **GPC**: `GPC_LEGALLY_REQUIRED_JURISDICTIONS = ["US-CA","US-CO","US-CT","US-VA"]`;
  GPC treated correctly as a *signal not a decision* (explicit user action
  overrides per W3C); per-category `respectGPC:false`; applies to all
  jurisdictions by default, scopable.
- Re-prompt on `jurisdictionChanged`.

So the opt-out regime is *partially* present via GPC, but the **opt-in vs
opt-out default-consent divergence — the actual legal heart of the question —
is missing**. This is the single most important feature to build for the
merge, and it's the natural payoff of unifying the two jurisdiction models:

**1.0 design — one jurisdiction concept, two axes.** OpenPolicy's
`jurisdictions` is the *legal/policy-text* axis (what the document must say).
OpenCookies' `Jurisdiction` + resolvers is the *runtime-behavior* axis (where
the visitor is). Unify them: the declared legal jurisdictions + the resolved
visitor jurisdiction together select a **default consent posture** that core
applies to the initial state:

- EEA / UK / CH / BR / AU / CA-style opt-in ⇒ non-essential default **off**,
  scripts gated until affirmative consent.
- `US` / `US-*` opt-out ⇒ non-essential default **on**, banner presents
  "Do Not Sell/Share" + honors GPC as the opt-out (GPC already wired).
- `ROW`/unknown ⇒ conservative opt-in default.

This belongs in **core** (a `jurisdictionPosture` resolver feeding the initial
`decisions`/`route`), driven by the same declaration that drives the rendered
policy — so the policy text ("EEA visitors: we ask consent before…") and the
banner behavior provably agree. It also retires the silent footgun where every
deployment is accidentally opt-in until the banner author hand-codes opt-out.

### 4.2.1 The jurisdiction capability table — the single seam

**One canonical `JurisdictionId`** — lowercase-kebab, legally-precise stem
(`eea`, not `eu`: GDPR applies EEA-wide). **One way to reference a jurisdiction
across the whole stack.** OP and OC both migrate to it — we do *not* keep two
enums plus a permanent bridge. The dual-enum was an artifact of two separate
repos; the merge removes the reason for it, so it goes. One table, read by
every axis, so adding a jurisdiction is **one row, not edits in four places**:

```ts
type JurisdictionId =
  | "eea"    // EEA — GDPR
  | "uk"     // UK GDPR + PECR
  | "ch"     // Switzerland — revFADP
  | "br"     // Brazil — LGPD
  | "ca"     // Canada — PIPEDA (+ Quebec Law 25 via sub-jurisdiction)
  | "us"     // US federal baseline — no omnibus law; opt-out by state
  | "us-ca"  // California — CCPA/CPRA
  | "us-co"  // Colorado — CPA
  | "us-ct"  // Connecticut — CTDPA
  | "us-va"  // Virginia — VCDPA
  | "row";   // Rest of world — conservative opt-in fallback

type JurisdictionCapability = {
  id: JurisdictionId;
  consentModel: "opt-in" | "opt-out";    // §4.2 posture axis
  policyText: "specific" | "equivalent"; // honesty flag — read by validator, not renderer
  parent?: JurisdictionId;               // sub-jurisdiction posture + text inheritance
  gpcLegallyBinding: boolean;            // GPC is always a *signal*; this is legal force
  /** migration-only: pre-1.0 OC codes (`EEA`,`US-CA`). Warn-and-map,
   *  removed once OC is migrated. NOT part of the frozen surface. */
  aliases?: readonly string[];
};

type JurisdictionTable = Readonly<Record<JurisdictionId, JurisdictionCapability>>;
```

Three pure consumers (reused verbatim by the hosted layer, §9):

- **Validator (§2.1):** every declared jurisdiction with
  `policyText:"equivalent"` emits a located `jurisdiction-generic-policy-text`
  `IssueCode`. This is what makes breadth *honest* instead of silent — and is
  precisely why we keep breadth rather than prune.
- **Posture resolver (§4.2):** reads `consentModel`; **composes** with
  per-category `lawfulBasis` (§4.1), does not duplicate it.
- **Resolution:** `US-${string}` (the 2025 state-law tail) falls back to its
  `parent` (`us`, opt-out) — breadth without enumerating the long tail.

Three support tiers, not a binary: `specific` (hand-authored), `equivalent`
(posture-correct + parent text + a suppressible diagnostic — **a legitimate,
shippable v1 tier**), absent (not in the union). **v1 ships exactly 11
`JurisdictionId`s (decided):** 3 `specific` — `eea`, `uk`, `us-ca` (matches
what OpenPolicy backs today) — and 8 `equivalent` — `ch`, `br`, `ca`, `us`,
`us-co`, `us-ct`, `us-va`, `row`. The union is frozen at these 11; the
`specific`/`equivalent` *tier* of any member can be upgraded post-1.0 without a
breaking change (it only removes a diagnostic), so promoting jurisdictions
later is non-breaking — the union membership is the frozen part, not the tier.

This collapses three plan items into one artifact: §2.1's prune decision
(now: keep breadth + diagnose), §4.1's enum-mismatch impedance (now: one id),
and §4.3's "explicit reconciliation layer" (now: nothing to reconcile).
`aliases` is the *only* migration concession — same one-time `console.warn`
mechanism as the §10 scope rename, shipped in the `0.x` consent window and
**deleted before the freeze**, so the frozen `JurisdictionId` is one clean
union.

### 4.3 Scanner unification — complementary, not conflicting

Both scanners use oxc-parser but scan for **different things**:

- OpenPolicy: **declarations** (`collecting()`, `thirdParty()`,
  `defineCookie()`) → codegen for the *policy doc*.
- OpenCookies: **ungated runtime usage** (`document.cookie`, `js-cookie`,
  `cookies-next`, `next/headers`, `Set-Cookie`, vendor imports/globals via
  `vendors.json`; gate-awareness via `<ConsentGate>`/`.has()`/helpers;
  suppression comments; `mode: warn|error`) → warnings only, no codegen.

They're complementary, and unified they unlock a differentiator OSS competitors
don't have: **cross-check declared vs actually-used.** "You gate `posthog` but
never declared it in your cookie policy" / "you declared `marketing` cookies
but no marketing tracker is gated." That bidirectional consistency check —
policy ↔ runtime — is a flagship PolicyStack feature.

Mechanically: one Vite plugin, **one oxc walk**, two rule sets, dual output
(on-disk gen module for the policy per §2.2 + warn/error reporter for ungated
usage + the autosync "missing category" suggestion). **Registry unification (corrects this doc's earlier
wrong claim that OpenPolicy lacks a vendor registry — Note / decision 13).**
There are *two complementary* registries today: OpenPolicy's
`packages/vite/src/known-packages.ts` (`KNOWN_PACKAGES` /
`KNOWN_COOKIE_PACKAGES`, keyed by **npm dependency name** → `ThirdPartyEntry` +
cookie categories) and OpenCookies' `vendors.json` (keyed by **import
specifier / runtime global / scriptUrl**). Merge them into **one canonical
vendor registry keyed by vendor**, carrying *both* lookup dimensions plus the
policy `ThirdPartyEntry` and implied cookie categories — so the policy side,
the ungated-usage scan, and the declared-vs-used cross-check all resolve
through one source. `usePackageJson` is **kept** (opt-in, both `thirdParties`
and `cookies` flavors) as the zero-annotation *detector* that reads
`package.json` deps against this registry — the dependency-presence analogue of
`collecting()`, and the §2.5 "technically knowable → inferred" path for third
parties. It stays **opt-in, not default-on**: dependency *presence* ≠ *use*
(server-only/unused deps), and over-disclosure in a privacy policy is its own
legal defect; the import/usage walk is the more precise truth, `usePackageJson`
the supplement for vendors that never appear syntactically. Diagnostics (§7.2)
label *which* detector found each vendor. **Decided: opt-in** — and the
deciding reason is broader than over-disclosure: explicit
`usePackageJson`/`thirdParty()`/`collecting()` are greppable and land in git
history, so the disclosure surface is itself **auditable**. Inference-by-
default would make "what do we disclose, and since when" invisible.

### 4.4 The banner UI / shadcn

OpenCookies is intentionally **headless — no UI, no shadcn registry**. The
"shadcn registry banner" lives on the OpenPolicy/policystack.dev side. The
full 1.0 story = the headless `@policystack/consent` core + the `./consent`
framework subpaths (§4.5) **+** an optional shadcn-registry styled banner
built on them (pre-built but fully overridable,
consistent with "consent logic, not consent UI"). Keep the registry banner out
of the frozen package API (it's a copy-in template, not an imported API).

### 4.5 Repo mechanics & the single per-framework package (decision 5)

Bring OpenCookies in under `packages/consent*`. The headless core stays its own
package — rename `@opencookies/*` core → **`@policystack/consent`** — but the
**per-framework consent adapters do not get their own packages**. They fold
into the single per-framework packages as a subpath export:

| Package | `./policy` | `./consent` |
|---|---|---|
| `@policystack/react`   | ✅ `PolicyDocument`, `PolicyComponents` | ✅ `ConsentProvider`, `useConsent`, `useCategory`, `ConsentGate` |
| `@policystack/vue`     | ✅ | ✅ |
| `@policystack/svelte`  | ✅ | ✅ |
| `@policystack/solid`   | — (no policy renderer exists) | ✅ |
| `@policystack/angular` | — | ✅ |

`@policystack/consent` (headless core) and `@policystack/scripts`
(framework-agnostic GA4/Meta/PostHog/Segment gating) stay standalone — they
aren't framework adapters, so they don't belong behind a framework package.

Rationale: package boundaries are **not** bundle boundaries — subpath `exports`
+ `sideEffects:false` give module-graph tree-shaking, so `@policystack/react/policy`
pulls zero consent code. One install / one version / one mental model is the
on-brand shape ("full consent *and* privacy stack") **and** a correctness gain:
since §4.5's fixed changeset group bumps both in lockstep anyway, two packages
would only add a version-skew failure mode (policy `@1.2` + consent `@1.1`, a
combination the §4.1 bridge makes silently broken). One package removes it.
Asymmetry (`solid`/`angular` are consent-only) is *more* coherent than the old
split, not less: "the framework package carries what exists for that framework."

Fold the OpenCookies scanner rules into `@policystack/vite`; move into
PolicyStack's **fixed** changeset group (switch from OpenCookies' independent
versioning); the unified `@policystack/cli` absorbs the (planned but unbuilt)
`scan`/`init`/`sync` ideas — no competing CLI to reconcile since it's an empty
scaffold.

### 4.6 Folding in policystack.dev — the site as dogfood

The site comes into the monorepo as an **app, not a package**: deployed, not
published; versioned with the repo, *not* in the changeset fixed group. Land it
as `apps/web` (the redirect shim stays `apps/www`).

The big upside: it already imports `@openpolicy/react`/`@openpolicy/sdk` from
npm to render its own `/privacy`. Inside the monorepo those become
`@policystack/*` `workspace:*` deps, so **the marketing site becomes a live
integration test** — a package regression breaks `/privacy` before publish.
Build order: packages → site. Its `.md`/`llms.txt` routes also make it the
natural home for the §8 agent-docs strategy (it's already half-built there).

The central tension is **toolchain divergence**. OpenPolicy/OpenCookies are
Vite+ (`vp`) + changesets + `packages/*`. policystack.dev is **plain Vite 8 +
TanStack Start + Nitro + content-collections**, with a bespoke
`vite build && node scripts/build-search-index.mjs` and TS 6 — it *cannot* use
`vp build` (that's the tsdown library build). **Resolution (decided): bring
the site onto the Vite+ catalog** like every other member — keep everything
consistent. It is still run via `vp run build` as a workspace script (the
documented escape hatch for non-`vp` scripts) and treated as an app target, not
a `vp pack` package. Consequence to accept up front: the site's bleeding-edge
Vite 8 / TS 6 / `nitro-nightly` get reconciled *down* to the workspace catalog
versions in phase 3 — "consistent" cuts both ways.

The sharp hazard: the root `pnpm-workspace.yaml` has
`overrides: { vite: "catalog:" }` pinning every workspace member to Vite+'s
vendored Vite. Forcing the TanStack Start app onto that build of Vite may
break the router plugin / Nitro / content-collections stack — still the single
biggest technical risk in the fold-in (§11). **Decided: align to the catalog,
not exempt.** Per-package override exemption is retained only as a *documented
contingency* if the stack genuinely cannot run on the catalog Vite after the
site's deps are reconciled down; it is not the plan. Phase 3 gates on the site
building green on the catalog Vite.

Deployment topology post-merge: **three surfaces, one repo** — (1) `apps/web`
(TanStack Start → Nitro → Vercel, policystack.dev domain), (2) `apps/www`
(redirect shim, openpolicy.sh domain), (3) the published `@policystack/*`
packages. Two Vercel projects (two domains) on one repo, differentiated by root
dir / build filter; the site build must depend on the packages building first.
Keep PolicyStack Cloud (hosted) *implementation* out of the monorepo — the site only
carries its *marketing copy*, which is fine and on-message.

## 11. Risks & open questions

- **Scanner delivery rewrite (§2.2/§7) is the riskiest single change** — right
  call, touches dev/build/SSR resolution. Sequence it *first*, behind the one
  retained canonical example + the scanner's oxc-walk unit tests (the matrix is
  gone, §12 phase 0), *before* the rename (one variable at a time).
- **The jurisdiction *posture model* (§4.2/§4.2.1) is the hardest design
  problem** — enum *reconciliation* is no longer a risk (§4.2.1 makes it one
  canonical id). It is the flagship feature: get the posture model right
  (legal axis × resolved axis × GPC interaction) and tested per-jurisdiction
  before freezing the consent API.
- **The canonical-id unification (§4.2.1) is technically breaking** — OC
  renames `EEA`→`eea` / `US-CA`→`us-ca`; OP keeps its codes but gains `eea`
  and the 6 previously-reserved codes become `equivalent`-tier (diagnosed, not
  removed). Net it's a *correctness* fix — call it out loudly in the changelog;
  the migration `aliases` shim softens it across the `0.x` window.
- **Bridge, don't heuristic.** The lossy `lawfulBasis==="legal" ⇒ locked`
  mapping must not ship; specify an explicit bridge type.
- **i18n scaling — resolved (§2.6, decision 14):** `Locale` stays closed +
  frozen with the exhaustive `Dictionary`; the post-1.0 scaling escape is a
  custom-`Dictionary` extension point (§9), not union-widening; OC's
  free-string `locale` is reconciled to one shared `Locale` (record stays
  `string` at rest for legacy tolerance), warn-and-map shim for the break.
- **The site's Vite toolchain vs the workspace `vite` override (§4.6) is the
  biggest fold-in risk.** `overrides: { vite: "catalog:" }` would force the
  TanStack Start app onto Vite+'s vendored Vite and likely break
  router/Nitro/content-collections. **Decided: align to the Vite+ catalog** for
  consistency — the site moves onto the vendored Vite and its bleeding-edge
  TS 6 / Vite 8 / `nitro-nightly` are reconciled *down* to workspace versions
  in phase 3. Residual risk: TanStack Start/Nitro may not run on the catalog
  Vite; phase-3 build-green gate, with per-package override exemption as the
  documented fallback only.
- **Angular adapter — decided (phase-3 spike, not resolved now).** It is
  `0.0.0`, unpublished, and uses `ng-packagr`. Attempt to migrate it onto
  `vp pack` like every other package so the monorepo has **no special-case
  build tooling**. If `vp pack` cannot cleanly build the Angular package,
  **drop Angular from 1.0** (revisit post-1.0 on demand) — there is no
  `ng-packagr` exception and no late-publish path. Flagged for phase 3.

### 13.2 One config — the single source of truth (§4.1)

```ts
// policy.config.ts
import { defineConfig } from "@policystack/sdk";
import { dataCollected, thirdParties, cookies } from "./policystack.gen"; // on-disk scanner output (§2.2/§7)

export default defineConfig({
  type: "privacy",
  company: { name: "Acme, Inc.", url: "https://acme.com", contact: "privacy@acme.com" },

  // canonical JurisdictionId (§4.2.1) — drives BOTH the policy text and the consent posture
  jurisdictions: ["eea", "uk", "us-ca"],

  data: {
    collected: { ...dataCollected },              // ← from scanner
    context: {
      // exhaustiveness enforced by defineConfig's generic — a scanned key with no
      // context here is a compile error, not a silent weak policy
      email:     { purpose: "Account & login",   lawfulBasis: "contract", retention: "Life of account", provision: "required" },
      analytics: { purpose: "Product analytics", lawfulBasis: "consent",  retention: "14 months",        provision: "optional" },
    },
  },

  cookies: {
    used: { ...cookies },                         // ← from scanner; { essential: true, … }
    context: {
      analytics: { lawfulBasis: "consent" },      // ← consent ⇒ this category is gated
      marketing: { lawfulBasis: "consent" },
    },
  },

  thirdParties: [...thirdParties],                // ← from scanner

  consentMechanism: { hasBanner: true, hasPreferencePanel: true, canWithdraw: true },
});
```

There is **no second "consent config."** `categories`, `locked`, the
opt-in/opt-out posture, and the re-prompt `policyVersion` are all *derived* from
this object (§4.1): `cookies.used` keys → categories, `essential:true` →
`locked`, `cookies.context[k].lawfulBasis` → whether the category is gated, the
compiled `version` hash → `policyVersion`.

Per §2.5 / decision 9 the `...dataCollected` / `...thirdParties` / `...cookies`
spreads above become **implicit** — `defineConfig` merges the gen module
itself — so the final v1 config is smaller still; the spreads are drawn here
only to show provenance, and `consentMechanism` is *derived* from the wired
runtime rather than written.

### 13.3 Wire the plugin once

```ts
// vite.config.ts
import { policyStack } from "@policystack/vite";

export default {
  plugins: [
    react(),
    policyStack({ config: "./policy.config.ts", scan: { mode: "warn" } }),
  ],
};
```

One oxc walk (§4.3) does three things: emits `policystack.gen.ts` (the data
above), warns on **ungated** cookie/vendor usage, and cross-checks **declared
vs actually used** ("you gate `posthog` but never declared it"; "you declared
`marketing` cookies but nothing marketing is gated").

### 13.4 Instrument the app at the call site

Annotate where data is actually touched — the scanner reads these syntactically
(§6, frozen call shapes):

```ts
import { collecting, thirdParty, defineCookie } from "@policystack/sdk";

function onSignup(form: SignupForm) {
  const email = collecting("email", form.email);  // → data.collected.email
  // …
}

defineCookie("analytics", "_ps_id");              // → cookies.used.analytics
thirdParty("posthog");                            // → thirdParties[]
```

These are the only edits that live in app code. The gen module + the typed
augmentation mean: collect something you didn't describe → **build-time type
error**, not a silently incomplete policy.

### 13.5 Render the policy

```tsx
// routes/privacy.tsx
import { PolicyDocument } from "@policystack/react/policy";
import policy from "../policy.config";

export default function Privacy() {
  return <PolicyDocument config={policy} />;   // slots overridable via PolicyComponents (§6)
}
```

This is exactly how policystack.dev renders its own `/privacy` (§4.6), so the
package is a live integration test.

### 13.6 The consent runtime — same `policy`, zero duplication

```tsx
// app root
import { ConsentProvider } from "@policystack/react/consent";
import policy from "./policy.config";

export function App({ children }) {
  return <ConsentProvider policy={policy}>{children}</ConsentProvider>;
}
```

`ConsentProvider` takes the **same config object**. From it, core (§4.2)
resolves the visitor's jurisdiction and applies the correct *default posture*:

- visitor in `eea` / `uk` → non-essential categories default **off**, scripts
  gated until affirmative consent;
- visitor in `us-ca` → opt-out: non-essential default **on**, "Do Not
  Sell/Share" + GPC honored as the opt-out;
- unknown / `row` → conservative opt-in.

The user writes none of that — it falls out of `jurisdictions` +
`cookies.context[k].lawfulBasis`.

Gating UI and scripts:

```tsx
import { ConsentGate, useCategory } from "@policystack/react/consent";

<ConsentGate category="analytics">
  <ProductAnalytics />
</ConsentGate>;

const marketing = useCategory("marketing"); // { granted, grant, deny }
```

```ts
// scripts that must not run until consent
import { gateScript } from "@policystack/scripts";
import { posthog } from "@policystack/scripts/posthog";

gateScript(posthog({ token: "phc_…", category: "analytics" }));
```

### 13.7 The banner (optional)

The copy-in shadcn banner is just a styled consumer of `useConsent()` — fully
editable, no API lock-in (§4.4). Or hand-roll one from
`useConsent()`/`useCategory()`; the legal behavior is already correct in core
regardless of the UI.

### 13.8 The agent loop

```sh
npx @policystack/cli validate --json   # structured issues + non-zero exit (§8.2)
npx @policystack/cli mcp               # MCP: validate_config, explain_jurisdiction, scan_ungated, … (§8.3)
```

### 13.9 What the user never has to write — the payoff

- opt-in vs opt-out default state per visitor (was: hand-coded in the banner,
  silently opt-in everywhere — §4.2);
- "policy changed → re-ask for consent" (automatic via the `version` hash →
  `policyVersion`);
- keeping the cookie policy and the banner in sync (the scanner's
  declared-vs-used cross-check fails the build on drift);
- per-jurisdiction policy prose (the capability table backs every declared
  jurisdiction, diagnosing the weak ones instead of shipping them silently —
  §4.2.1).

One declaration; the policy text and the runtime behavior **provably agree**
because they are the same object.
